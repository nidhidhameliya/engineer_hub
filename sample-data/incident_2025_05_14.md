# Incident Report — Production Outage

**Incident ID:** INC-2025-05-14-001  
**Date:** May 14, 2025  
**Duration:** 2h 14m (14:23 UTC – 16:37 UTC)  
**Severity:** SEV-1 (Complete service degradation)  
**Status:** Resolved  

---

## Summary

The Payment Service experienced a complete outage for 2 hours and 14 minutes due to a Redis connection pool exhaustion caused by a missing connection timeout configuration in the v3.2.0 deployment. All payment processing was halted. Approximately 8,400 transactions were queued and processed within 45 minutes of service recovery.

---

## Impact

- **Users affected:** ~42,000 (all users attempting payment during the window)
- **Revenue impact:** ~$180,000 in delayed transactions (all processed post-recovery)
- **Services affected:**
  - Payment Service (primary)
  - Order Service (dependent — orders stuck in PENDING state)
  - Billing Service (dependent — invoices not generated)
  - Notification Service (secondary — payment confirmation emails delayed)

## Root Cause Analysis

### Primary Cause

The v3.2.0 release of the Payment Service introduced a new Redis caching layer for payment idempotency keys. The Redis client configuration **omitted the `connect_timeout` and `socket_timeout` parameters**.

When Redis experienced elevated latency (due to a separate memory pressure event from a bulk cache population job running at 14:20), payment service requests began hanging indefinitely, exhausting the connection pool (max: 50 connections).

**Offending code (payment-service v3.2.0):**
```python
redis_client = redis.Redis(
    host=REDIS_HOST,
    port=6379,
    db=0,
    # MISSING: connect_timeout=2, socket_timeout=2
)
```

### Contributing Factors

1. **Cache bulk load job** ran at 14:20 (5 minutes before deployment), causing Redis memory pressure
2. **Missing timeouts** — no circuit breaker on the Redis client
3. **Incomplete rollback procedure** — deployment included a database migration not covered in rollback runbook
4. **Alert threshold** — payment error rate alert set to 5% threshold, fired 12 minutes after first failures

---

## What Worked Well

- PagerDuty alert fired within 12 minutes
- Team communication in #incident-2025-05-14 was clear and organized
- Queued transactions were all successfully recovered after service restoration
- No data was lost

---

## What Didn't Work

- No staging environment testing of the new Redis integration under load
- Rollback procedure did not account for database migrations
- Redis connection pool exhaustion not caught in pre-deployment health checks
- Bulk cache job scheduling not coordinated with deployments

## Lessons Learned

1. **Always configure timeouts** on all external service clients (Redis, databases, HTTP calls)
2. **Rollback procedures must be tested** end-to-end, including database state
3. **Coordinate bulk jobs** with the deployment pipeline to avoid timing conflicts
4. **Circuit breakers** should be standard in all payment-critical service integrations
